# Tech Article Notification - 開発指示

## プロジェクト概要

Qiitaのトレンド人気記事5つを毎日取得し、Slackで通知しつつNotionで記事を管理するアプリケーション。

## コア要件

1. **Qiita連携**: トレンド上位5記事の毎日自動取得
2. **AI要約**: 記事から100字程度の紹介文を生成
3. **Notion統合**: 記事をNotionデータベースに蓄積
4. **Slack通知**: 毎日決まった時間に要約と共にSlack通知
5. **管理機能**: Notionで読了状況、評価、リピート読みの管理

## 技術スタック方針

### バックエンド
- **言語**: Node.js (TypeScript推奨) または Python
- **スケジューリング**: node-cron または APScheduler
- **Web スクレイピング**: cheerio または BeautifulSoup

### 外部API
- **Qiita**: 公開API または Web スクレイピング
- **Notion**: 公式 API (`@notionhq/client`)
- **Slack**: Bolt for Node.js または Python
- **AI 要約**: OpenAI GPT API (或いはローカルモデル)

### デプロイ
- **本番環境**: AWS Lambda + CloudWatch Events (スケジューリング)
- **ローカル開発**: Docker Compose (Redis, 他必要なら)

## コード品質基準

- **TypeScript**: 型チェック厳密モード (strict: true)
- **Linting**: ESLint + Prettier
- **テスト**: Jest で最小 70% カバレッジ
- **エラーハンドリ**: Try-catch + 適切なログ記録
- **ドキュメント**: 各Agent に README.md 記載

## Agentの責務分離

各Agent は**単一責任の原則**に従う:
- qiita-fetcher: 記事取得に専念、要約しない
- summarizer: 要約に専念、外部APIは他Agent に委譲
- notion-writer: Notion書き込み に専念
- slack-notifier: Slack送信に専念
- orchestrator: フロー管理に専念、具体処理は各Agent に委譲

## 注意事項

1. **API利用制限**: Qiita, Notion, Slack の制限を考慮（レート制限対策）
2. **AI要約精度**: OpenAI API の費用 vs 精度 をトレードオフ検討
3. **Notion スキーマ**: 後の変更を想定し拡張可能な設計
4. **スケーラビリティ**: 将来的に複数チャネル(複数Slackチャネル、メール等)対応を視野
